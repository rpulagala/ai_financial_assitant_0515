Project: AI Financial Assistant

Summary
# Developer-Oriented Summary — AI Financial Assistant for Local Governments

## 1. Product Objective

The project is to build a sovereign, secure, vertical AI financial assistant for local governments, including municipalities, inter-municipal bodies, and other public local authorities.

The goal is not to replace existing public finance software. The product must act as an intelligence layer above existing systems. It should allow users to import or connect financial data, structure it, query it in natural language, detect anomalies, generate dashboards, and produce usable reports or decision notes.

The assistant must help finance teams, chief administrative officers, elected officials, service managers, management controllers, consultants, and outsourced finance directors understand and exploit financial data more easily.

The product must provide reliable, sourced, explainable, and actionable answers. Every figure must be traceable to a source. Every calculation must be explainable. If the data are missing, incomplete, contradictory, or outside the user’s access rights, the assistant must refuse to answer or clearly state its limits.

The solution should start with a prototype based on Excel/CSV imports, then evolve into a web MVP, then into a scalable SaaS or private deployment architecture with connectors, APIs, advanced dashboards, RAG, and reporting features.

---

## 2. Core User Problem

Local governments already have a lot of financial data, but this data is often scattered across several tools and file formats. Finance teams frequently need to export Excel files, clean data, reconcile tables, manually check inconsistencies, and produce notes under time pressure.

The main operational problems to solve are:

1. difficulty obtaining a consolidated financial view quickly;
2. dependence on Excel exports and manual reprocessing;
3. difficulty explaining differences between voted budget, opened credits, committed amounts, mandated amounts, paid amounts, and available credits;
4. lack of early warning alerts on budget risks;
5. time-consuming production of notes for executives and elected officials;
6. poor readability of financial data for service managers and non-financial users;
7. heterogeneous software and data formats;
8. weak internal control tooling.

The product must therefore provide a practical interface between raw financial data and business decision-making.

---

## 3. Product Positioning for Development

The product must be designed as a vertical business assistant, not a generic chatbot.

A generic chatbot can generate plausible text. This product must:

1. use real internal data;
2. cite the sources used;
3. explain calculations;
4. distinguish facts, analysis, and recommendations;
5. refuse unsupported answers;
6. apply local public finance business rules;
7. respect user permissions;
8. isolate each local authority’s data;
9. generate usable deliverables;
10. maintain logs and audit trails.

For developers, this means the application must not be built around the LLM alone. The core system must include:

* a structured data model;
* an ingestion and mapping engine;
* a rules engine;
* a document indexing and retrieval system;
* a controlled AI orchestration layer;
* dashboards;
* export generation;
* user and permission management;
* logging and observability.

The LLM should be treated as one component within a controlled architecture.

## 5. Main User Profiles to Support

The system should support multiple user profiles, but the MVP can prioritize finance directors, chief administrative officers, finance agents, and service managers.

### Finance Director / Finance Manager

Needs advanced financial analysis, dashboards, alerts, reliable calculations, and fast note generation.

Typical questions:

* What is the current budget execution rate?
* Which chapters are drifting?
* Which commitments remain open?
* Which credits may be insufficient by year-end?
* Which suppliers concentrate anomalies?

Main development requirements:

* high data accuracy;
* correct distinction between committed, mandated, paid, and available amounts;
* source traceability;
* exportable analysis.

### Chief Administrative Officer

Needs executive summaries, risk prioritization, decision notes, and financial trajectory visibility.

Main development requirements:

* synthetic view;
* risk scoring or risk ranking;
* clear recommendations with limits;
* separation between confirmed data and interpretation.

### Elected Official

Needs readable, non-technical explanations.

Main development requirements:

* pedagogical reformulation;
* simplified notes;
* charts and key figures;
* no excessive technical vocabulary.

### Service Manager

Needs to track budget availability and commitments within their own scope.

Main development requirements:

* service-level dashboards;
* strict access rights;
* available credit calculation;
* alerts on overconsumption or open commitments.

### Finance Agent

Needs operational controls and anomaly detection.

Main development requirements:

* filtering;
* duplicate detection;
* old commitments;
* rejected mandates;
* imputation inconsistencies;
* exportable control lists.

### External Consultant / Outsourced Finance Director

Needs multi-client diagnostic capabilities.

Main development requirements:

* strict tenant isolation;
* report generation;
* reusable diagnostic templates;
* data source tracking.

## 6. Core Use Cases to Implement First

The MVP should not try to implement all use cases immediately. The first technical version should focus on a limited set of high-value use cases.

### Budget execution

The system should answer:

* What is the operating budget execution rate?
* What is the investment budget execution rate?
* Which chapters show abnormal consumption?
* Which articles increased the most compared with the previous year?
* Which credits are at risk of being insufficient before year-end?
* Which credits are under-consumed?
* What are the differences between voted, opened, committed, mandated, paid, and available amounts?

### Commitments

The system should detect:

* outstanding commitments;
* commitments older than six months;
* commitments linked to nearly exhausted credits;
* possible duplicate commitments;
* commitments without an associated public contract;
* services concentrating open commitments.

### Mandates

The system should analyze:

* rejected mandates;
* recurring rejection reasons;
* suppliers concentrating rejections;
* late mandates;
* mandates above a defined threshold;
* possible imputation anomalies.

### Suppliers

The system should identify:

* supplier concentration;
* duplicate third-party records;
* suppliers with abnormal spending volume;
* suppliers linked to anomalies;
* suppliers without contract reference where expected.

### Internal control

The system should provide:

* priority anomalies;
* control lists;
* risk labels;
* simple recommendations;
* exportable action plans.

## 9. Business Rules Engine

The business rules engine must detect financial risks, anomalies, and control points.

The MVP should include simple, explainable rules before adding advanced scoring.

### Priority rules

The rules engine should detect:

1. insufficient credits;
2. abnormal consumption;
3. under-consumption;
4. old outstanding commitments;
5. commitments not fully mandated;
6. commitments with no public contract reference;
7. rejected mandates;
8. suppliers concentrating large spending;
9. supplier dependency;
10. duplicate suppliers;
11. imputation inconsistencies;
12. late mandates;
13. old revenue titles if data exist;
14. general expenditure drift;
15. payroll drift if data exist.

### Required output for each alert

Each alert must include:

* rule name;
* rule category;
* severity;
* affected entity;
* affected lines;
* calculation;
* source;
* explanation;
* confidence level;
* recommendation;
* status.

### Example alert

Alert type: Old outstanding commitment
Severity: Medium or High
Condition: commitment date older than six months and remaining amount greater than threshold
Explanation: “This commitment remains open more than six months after creation and still has a remaining amount of €X.”
Source: commitment file, line X
Recommendation: “Verify whether the service has received the goods/services, whether the invoice is pending, or whether the commitment should be closed.”

The rule must be deterministic and auditable. The AI may explain the alert, but the detection should initially come from explicit rules.

## 10. Conversational AI Module

The conversational AI module must allow users to ask business questions in natural language.

It must support:

* business-oriented chat;
* intent classification;
* retrieval of structured data;
* retrieval of documents through RAG;
* sourced answers;
* calculation explanations;
* refusal when data are unavailable;
* reformulation for different user profiles;
* generation of notes;
* generation of action plans;
* response export;
* conversation history.

### Required answer structure

A recommended answer should contain:

1. short answer;
2. key figures;
3. explanation;
4. sources;
5. limits;
6. cautious recommendation;
7. possible next actions.

### Mandatory guardrails

The assistant must:

* never provide a figure without a source;
* never invent a document;
* never invent missing data;
* explicitly state missing data;
* distinguish fact, analysis, and recommendation;
* include calculation details;
* cite source files or documents;
* include a confidence level;
* refuse out-of-scope requests;
* respect user permissions;
* log prompts and responses.

### Developer implication

The LLM should not directly query raw databases without control. A safer architecture is:

1. classify user intent;
2. identify required data;
3. check permissions;
4. retrieve structured data through controlled queries;
5. retrieve relevant documents through RAG if needed;
6. compute figures through deterministic functions;
7. pass summarized structured context to the LLM;
8. generate answer with strict instructions;
9. validate that the answer contains sources;
10. log the full interaction.

## 11. RAG and Document Management

The RAG module must allow the assistant to answer questions based on uploaded documents.

Documents to index may include:

* budgets;
* budget amendments;
* administrative accounts;
* financial statements;
* budget orientation reports;
* deliberations;
* public contracts;
* agreements;
* supporting documents;
* internal notes;
* procedures;
* M57 documentation;
* audit reports.

### RAG requirements

The RAG system must include:

* indexing by local authority;
* document metadata;
* access-right filtering;
* business-aware chunking;
* hybrid search;
* vector search;
* keyword search;
* citations;
* minimum relevance threshold;
* contradiction handling;
* source links;
* document versioning.

### Mandatory metadata

Each document should include:

* document id;
* local authority id;
* fiscal year;
* document type;
* title;
* upload date;
* uploader;
* source file;
* confidentiality level;
* service scope;
* version;
* indexed status.

### RAG answer behavior

If the retrieved evidence is weak, the assistant must say so. If documents contradict each other, it must identify the contradiction and avoid giving a definitive answer unless the hierarchy of documents is clear.

The assistant must never use documents from another local authority or another unauthorized service scope.

## 16. AI Architecture

The AI layer should be designed as an orchestrated system, not a direct LLM interface.

### Main components

1. intent classifier;
2. permission checker;
3. structured data retriever;
4. document retriever;
5. calculation engine;
6. prompt builder;
7. LLM response generator;
8. response validator;
9. citation checker;
10. logging module.

### AI options

The preferred model provider is Mistral AI or another European AI provider.

Possible deployment strategies:

* API-based for prototype and MVP;
* European or sovereign cloud for public-sector acceptability;
* open-weight model for stronger control;
* private hosting for sensitive deployments.

The architecture should avoid model lock-in. The LLM provider should be abstracted through a model interface.

### Anti-hallucination rules

The system must enforce:

* no figure without source;
* no invented source;
* no answer if data are unavailable;
* explicit limits;
* calculation details;
* clear confidence level;
* separate facts, analysis, recommendations;
* refusal outside scope;
* logs for audit;
* evaluation on test cases.

## 18. Interoperability Strategy

The product must not depend on immediate API access from existing financial software vendors.

Integration should be progressive.

### Level 1 — Manual import

Users upload Excel or CSV files.

This is the MVP baseline.

### Level 2 — Automated file deposit

Files are periodically deposited in a secure space.

### Level 3 — Semi-automated connector

The system recognizes recurring formats from specific vendors.

### Level 4 — Partner API

Integration through vendor APIs when available.

### Level 5 — Native integration

The product becomes integrated with, or distributed alongside, existing business software.

The MVP should prioritize:

* budget;

* rejected mandates if available.

## 25. Key Technical Principles

The product must follow several non-negotiable principles.

### Source-first architecture

Every number must be linked to a source file, source row, source document, or computed formula.

### Deterministic calculations

Important financial calculations should be performed by backend services, not invented by the LLM.

### AI as explanation layer

The LLM should explain, summarize, reformulate, and draft notes. It should not be the only place where business logic lives.

### Strict tenant isolation

Every request, query, vector search, document access, and export must be filtered by local authority and user permissions.

### No silent failure

If data are missing, the system must say so. If an import is partial, the system must show warnings. If the AI lacks evidence, it must refuse or qualify the answer.

### Auditability

The product must be explainable after the fact: who uploaded what, which data were used, which rule generated which alert, which source supported which answer.

## 26. Final Developer Summary

The product to build is a secure vertical financial intelligence platform for local governments.

The first version must allow users to upload Excel/CSV files, map them to a normalized financial data model, compute budget and accounting indicators, detect simple anomalies, query data through an AI assistant, display dashboards, and generate sourced reports.

The most important modules are:

* ingestion;
* mapping;
* financial data model;
* rules engine;
* AI orchestration;
* RAG;
* dashboards;
* exports;
* user roles;
* tenant isolation;
* logging.

The developer should prioritize correctness, traceability, and security over advanced AI sophistication.

The MVP should prove that the system can answer practical questions such as:

* What is the budget execution rate?
* Which credits may be insufficient?
* Which commitments are old and still open?
* Which suppliers concentrate spending or anomalies?
* Which mandates were rejected?
* Can the system generate a sourced note for management or elected officials?

The long-term product can later add connectors, APIs, forecasting, advanced scoring, debt, treasury, procurement, and broader public finance analysis.

But the first technical objective is clear: build a reliable, secure, source-based assistant that transforms imported local government finance data into dashboards, alerts, explanations, and reports.
